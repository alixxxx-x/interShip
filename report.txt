PURPOSE & DESCRIPTION
---------------------
interShip! is a University-Enterprise Internship Management Platform designed to
fully digitize and automate the internship placement process in Algeria.

It connects three main actors:
  1. Students   — Apply for internships using a digital CV
  2. Companies  — Post offers and manage candidate pipelines
  3. Admins     — Validate placements and auto-generate official "Convention de Stage" PDFs

The system supports full lifecycle management:
  Offer Creation → Student Application → Company Review →
  Admin Validation → Automatic PDF Agreement Generation (Convention de Stage)

Students MUST register using an @univ.dz university email.

DATABASE MODELS (apis/models.py)
---------------------------------

1. User (AbstractUser)
   - email (unique, used as USERNAME_FIELD)
   - username (nullable)
   - role: STUDENT | COMPANY | ADMIN_DEPT | ADMIN_UNIV
   - profile_picture (ImageField)
   Uses Multi-Table Inheritance (MTI) — each role maps to its own table:

2. Student (extends User)
   - university_id, wilaya, phone, major

3. Company (extends User)
   - name, logo, description, location, website, company_field, founded_year

4. Administrator (extends User)
   - department

5. AdminUniv (extends User)
   - university_name, departments (JSONField — list of departments)

6. InternshipOffer
   - title, description, company (FK→Company)
   - status: DRAFT | OPEN_FOR_APPLICATION | CLOSED_FOR_APPLICATION |
             ONGOING | FINISHED | CANCELLED | ARCHIVED | HIDDEN
   - internship_location: REMOTE | ONSITE | HYBRID
   - internship_type: FULL_TIME | PART_TIME
   - internship_structure: FOR_CREDIT | CO_OP | FELLOWSHIP
   - offer_start_date, offer_end_date, internship_duration (auto-calculated)
   - number_of_places, internship_salary, internship_skills
   - internship_image, wilaya

7. Application
   - student (FK→Student), internship (FK→InternshipOffer)
   - status: PENDING | ACCEPTED | REJECTED
   - application_date (auto), is_validated_by_admin (bool), admin_validation_date
   Business Logic:
     - On save/delete: triggers update_internship_status()
     - When admin-validated acceptances reach number_of_places → auto-CLOSE internship
       + bulk-REJECT all remaining unvalidated applications + send notifications
     - When count drops below limit → auto-REOPEN internship

8. Skills
   - name, skill_level: BEGINNER | INTERMEDIATE | ADVANCED
   - FK to InternshipOffer

9. DigitalCV (OneToOne→Student)
   - first_name, last_name, email, phone, linkedin, github, portfolio_link
   - skills, experience, education, profile_summary, address, languages
   - wilaya, university_id, date_of_birth, nationality
   - image (ImageField), cv_file (FileField — uploaded PDF CV)
   - created_at, updated_at

10. Notification
    - recipient (FK→User), notification_type, message, is_read, application (FK nullable)
    - Types: NEW_APPLICATION | APPLICATION_ACCEPTED | APPLICATION_REJECTED |
             VALIDATION_REQUIRED | NEW_INTERNSHIP_FROM_FOLLOWED
    - Ordered by -created_at

11. CompanyFollow
    - student (FK→Student), company (FK→Company)
    - followed_at (auto)
    - unique_together: (student, company)

12. Message
    - sender (FK→User), recipient (FK→User), content, is_read, created_at

13. PasswordReset
    - user (FK→User), code (6-char), created_at, is_used

JWT / AUTH FLOW
---------------
1. User logs in → POST /api/auth/login/ → receives access + refresh tokens
2. Tokens stored in localStorage:
     access  → key: 'banana'
     refresh → key: 'katkouti_hayalabib'
3. Axios request interceptor: reads access token → adds Authorization: Bearer <token> header
4. Axios response interceptor: if 401 received:
     a. Gets refresh token from localStorage
     b. POSTs to /api/auth/refresh/
     c. Stores new access token
     d. Retries original request
     e. If refresh also fails → clear localStorage → redirect to /login
5. ProtectedRoute: decodes refresh token with jwt-decode → if expired → redirect to /login

================================================================================
  KEY FEATURES & WORKFLOWS
================================================================================

1. STUDENT REGISTRATION & CV
   - Must register with @univ.dz email (backend-enforced)
   - Multi-step CV builder modal (CreateCvModal.jsx — stepper UI)
   - Can upload a PDF CV file OR use built-in CV builder
   - CV stored as DigitalCV model (OneToOne with Student)
   - Student CANNOT apply without having a CV

2. INTERNSHIP SEARCH (Internships.jsx — ~35KB)
   - Search by title, company, keywords
   - Filters: type (Full/Part Time), location (Remote/Onsite/Hybrid), wilaya, skills
   - Shows followed-company internships feed
   - Countdown timer on deadlines

3. APPLICATION WORKFLOW
   - Student applies → status=PENDING → notification sent to Company
   - Company accepts → status=ACCEPTED → notification to Student + to all Admins
   - Company rejects → status=REJECTED → notification to Student
   - Admin validates → is_validated_by_admin=True → notification to Student + Company
   - Admin can generate "Convention de Stage" PDF after validation
   - If spots fill up → internship auto-closes → remaining applicants auto-rejected

4. COMPANY FOLLOW SYSTEM
   - Students can follow/unfollow companies
   - When a followed company posts a new OPEN internship → followers get notifications
   - CompanyDetails page shows follow/unfollow button with live status

5. MESSAGING SYSTEM
   - Direct messaging between any users
   - Conversation-style view (filtered by recipient_id)
   - Marks messages as read when conversation is opened
   - ChatModal for Threads-style UI

6. NOTIFICATION SYSTEM
   - Real-time badge in Navbar (dot indicator + count)
   - Notification panel accessible from navbar dropdown (modal overlay)
   - Notification types: new application, accepted, rejected, validation required, new internship from followed
   - Mark individual / mark all / clear all
   - Emits custom DOM event 'notificationsUpdated' to sync badge count across components

7. AI CHATBOT (Google Gemini)
   - Accessible on FAQ page
   - Chat history preserved in session (sent with each request)
   - Gemini model: gemini-2.0-flash
   - System prompt: internship platform assistant, friendly tone
   - Student role: informed about @univ.dz requirement
   - Company role: corporate recruiter tone

8. PDF GENERATION (Backend ReportLab)
   a. Convention de Stage (Internship Agreement):
      - Only generated after admin validates application
      - Contains: student name, company, internship title, duration, start date, validation date
      - Has signature fields: Student / Company / University
      - Downloadable via GET /api/admin/applications/<id>/agreement/
   b. Student CV PDF:
      - Generated on demand via GET /api/cv/generate/<student_id>/
      - Formatted A4 layout with sections: contact info, profile, skills, experience, education

9. ADMIN VALIDATION QUEUE
   - Shows only applications where: status=ACCEPTED AND is_validated_by_admin=False
   - Admin can validate → generates notifications to student + company
   - Admin can reject → reverts to REJECTED, notifies both parties

10. SETTINGS PAGE (6 tabs)
    - Account: update profile picture, name, phone, wilaya, company info, etc.
    - Security: change password (validates old password)
    - Appearance: toggle dark/light mode
    - Language: switch EN/FR/AR (all strings translated via language-provider.jsx)
    - Privacy Policy: static content
    - Support Center: links to help docs

11. MULTI-LANGUAGE SUPPORT (EN/FR/AR)
    - Context-based via LanguageProvider + useLanguage() hook
    - language-provider.jsx contains ~75KB of translation strings
    - All UI strings use t("key") pattern
    - Language preference stored in localStorage
